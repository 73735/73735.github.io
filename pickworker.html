Mujin Pick Worker®

Making industrial robots smarter.

Problem

Previously, programming an industrial robot required defining the entire motion path of the movement. The technology behind Pick Worker™ is capable of automatically generating motion paths for industrial robots in real time. API of Pick Worker® is optimized for bin-picking application and the user interface to teach what to pick and where to place is needed. The interface must work well on desktop as well as industry standard resistive touch screen.

Requirements

After analysing a scenario run via command line and manually constructed user config file, we determined that tasks needed to be performed via GUI are configuring the network and project, modeling and configuring the modules, modeling the environment, defining the robot and its tool coordination, calibrating 3D sensors, defining how to grab a target and how to place the target.

Users

After interviewing stakeholders, we identified two user groups to which our software needs to be optimized for.

Operator

Operators are the end user of the application. They usually have minimum training and no engineering knowledge. Their main task is to start and stop the pick-and-place cycle as needed. They also need to be able to look into the log and report what the error messages are when the system stopped unexpectedly. They prefer simple and easy to understand interface.

System Integrator

System Integrators are those who set up both hardware and software of the bin-picking system. They are usually educated to the Bachelor's level and beyond in electrical engineering or mechanical engineering and went through extensive trainings. They are the core user of the application and expected to read the user manual carefully. They prefer quick-to-operate interface. They are also familiar with pre-existing CAD softwares such as Autodesk Inventor and PTC Creo.

Information Architecture

In creating Schematic Diagram, procedure-based categorization was used first in order to hide unnecessary items in each step and also clarify the concept of how the system works and what steps are needed. Then component-based categorization was used in order to further divide tasks. Installation procedures were divided into ‘modeling’, ‘goal editing’, ‘calibration’, ‘picking parameters settings’, ‘detection settings’ ’and ‘simulation’. Under each procedure, components were defined as ‘robot’, ‘gripper’, ‘sensor’, ‘container’, ‘plc’, ‘work’ and ‘layout’.

Prototype

Clickable mid-fidelity prototype was created using Balsamiq. Fixed header and footer were used in order to inform users the current location within the interface and the status of the system. Collapsable panels commonly seen in standalone CAD software were used in order to make the most of the screen space which is 1240x600 (7 inch).

Icons

I designed s set of icons using simple geometical shapes and a single color so that icon's color can be changed programatically.

Solution

We decided to create two separate modes ‘standard (SD)’ and ‘system integrator (SI)’ so that the operators does not need to understand the unneeded features also preventing them from accidentally changing the set up. Due to the complexity of the application and concept, a wizard mode was added to train system integrators.

Set Up

The welcome screen appears when there is no project is configured.

Modeling

The modules and environment used for motion planning will be created using the model editor. The modeling process involves configuration, modeling of modules, layout and modeling of works.

Configuration

The modeling starts with a configuration phase. First, the system network needs to be set up.

Next, the scope of the project needs to be determined using either a wizard-style interface or regular configuration screens.

Once the configuration phase is finished, users are presented with the modeling menu showing the modeling category and the number of remaining tasks.

Robot

The robot needs to be modeled first. It can be selected from the library or uploaded via USB drive using Collada file format.

Link

A link is what connects hierarchical geometries. It functions as a hinge containing the local rotation axis information of am object.

Joint

A joint is the physical shape of a robot section nested under a link. Non-root joints contain minimum and maximum joint values along with other parameters.

Gripper

A gripper is used to grasp a target. The grasp type and other advanced parameters can be set here.

After the gripper has been modeled, it needs to be attached to a link of the robot using the 'add gripper' icon on the robot modeling page.

Tool

A tool is the coordinate of the gripper used to grasp the target work. The 'tool tab' will be available once a gripper is added.

Frame

A frame is a moving part of the hierarchy of the robot. The root and tip links need to be defined along with the origin of the gripper in order to function properly.

Sensor

A Sensor, a camera container, is a collection of cameras which together function as a 3D sensor. The sensor ID and network information need to match with the physical environment.

Camera

Cameras collectively function as a 3D sensor. The type of cameras usually includes left, right and color.

Database

The database contains a list of models that can be imported into the scene. New models can be uploaded via USB or imported from the library.

Container

A container is a premade geometry that can function as a source and destination in the project. Paddings, margins and cross sections can be added easily.

Various parameters for point cloud generation and collision avoidance can be set here.

PLC

PLC (Programmable Logic Controller) controls inputs and outputs. Although they do not physically exist in the scene, they need to be edited here as they correspond to devices in the scene.

Layout

A layout is a scene containing all the scenes of modules. Once all the modules were modeled, they need to be organized in the layout along with the props that exist in the real environment such as walls and ceilings.

The structure of an object in the scene consists of the base link containing geometries and meshes. Each geometry and mesh has its own local transformations, rotations, and scales along with drawing properties.

Work

A work is a target object for the robot to grasp. An accurate Collada file needs to be uploaded for CAD matching detection.

Grasp Point

A grasp point is a coordinate used to grasp the work and needs to be defined by users. Various coordinate systems will be used depending on the type of gripper and the shape of the work. Furthermore, points can be organized into sets.

Goal

Where the target works need to be placed will be defined using the goal editor.

Goal Point

A goal point is a coordinate where works are placed by the robot. Various coordinate systems will be used depending on how the work needs to be placed. Furthermore, points can be organized into sets.

Goal Order

Goal order is a full sequence of placement. Each goal contains one or more goal points.

The world space coordinate system can be used when the works are placed on stationary locations. Any position in the scene can be stored.

The container space coordinate system can be defined for each container. The relative positions to the origin of the container will be stored.

A valid grasp set can be defined for each goal or each goal point in order to specify what can be placed based on how the work is grasped.

All the possible solutions for a selected goal point can be displayed by pressing the "solution" icon.

Calibration

Sensors and tools need to be calibrated in the physical environment. The process involves moving the robot with a calibration pattern in various predefined paths. The calibration results are need to be verified as needed.

Exposure and color scale need to be adjusted before starting the calibration cycle.

Calibration parameters and calibration methods for each camera need to be set.

Once calibration is finished, the results will be automatically displayed prompting users to decide whether to apply the new results.

Users can verify the calibration results using the quicker verification process.

Picking Parameters

Picking parameters are generated after the configuration and are organized into the four phases of the cycle: pick, transit, place and post cycle execution.

Viewer

Using collapsible panels, the viewer can be expanded or contracted depending on the need.

Operation

The operation screen becomes the home screen once the project is configured. The screen is mainly used by operators with minimum training that the interface features a simple conceptual model similar to that of a music player.
